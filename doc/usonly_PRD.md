## 产品概述

**产品名称：** UsOnly（暂定）

**产品形态：** 网页应用（响应式设计，支持移动端）

**一句话描述：** 一个只属于两个人的私密空间，用轻松自然的方式分享彼此的日常生活。

**核心价值：**
- **私密 (Privacy)**：默认仅两人可见，安全放心
- **自然 (Natural)**：无压力参与，想分享就分享
- **连接 (Connection)**：通过日常碎片，感受彼此的存在

**产品定位说明：**
UsOnly 是一个只属于你们两个人的私密空间。为想要记录日常却不想公开发布到社交媒体的情侣设计——没有他人的点赞和评论，只有你们彼此分享的真实瞬间。不需要精心修图、写文案，想发就发，随心记录。

## 目标用户

**主要人群：**
- 异地恋情侣
- 工作忙碌但想保持连接的伴侣
- 喜欢记录生活但不想公开发布到社交媒体的人

**用户特征：**
- 重视隐私
- 想要记录关系中的美好瞬间
- 不希望记录变成负担或任务
- 不想在社交媒体上经营人设，只想做真实的自己

## 核心理念

### 设计原则

| 原则 | 说明 |
|------|------|
| **私密优先** | 默认仅两人可见，公开需要主动选择 |
| **无压力参与** | 没有强制打卡、没有连续记录要求 |
| **差异即趣味** | 不要求内容匹配，不同视角才是亮点 |
| **轻量互动** | 操作简单，30 秒内完成一次分享 |

### 与社交媒体的区别

| 维度 | 社交媒体 (朋友圈/Instagram) | UsOnly |
|------|---------------------------|--------|
| 可见性 | 公开给所有人/好友 | 仅两人可见 |
| 内容要求 | 精心修图、写文案 | 随性分享，真实就好 |
| 互动方式 | 点赞、评论、转发 | 无点赞评论，只有彼此 |
| 心理压力 | 追求完美形象 | 放松做真实自己 |
| 观众 | 多人/朋友/关注者 | 只有 TA 一人 |

### 与传统记录产品的区别

| 维度 | 传统记录产品 | UsOnly |
|------|-------------|--------|
| 可见性 | 默认公开/个人可见 | 默认仅两人可见 |
| 参与方式 | 单人记录 | 双人呼应式记录 |
| 内容要求 | 完整、精美 | 真实、随性 |
| 互动压力 | 需要评论/点赞 | 无需回应，存在即连接 |

## MVP 功能范围

### 第一版只做一件事

**「平行视角」每日分享**

系统每天提供一个轻松主题，双方各自随时上传照片或文字，系统自动并排展示，形成"我们的今日"。

### 功能清单

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 用户注册/登录 | P0 | 基础账号系统 |
| 情侣配对 | P0 | 通过邀请码绑定两人关系 |
| 每日主题 | P0 | 系统提供一个开放主题 |
| 上传分享 | P0 | 支持照片 + 文字（可选） |
| 时间轴展示 | P0 | 按时间顺序展示所有分享 |
| 私密设置 | P0 | 默认私密，无公开功能（第一版不做） |
| 通知提醒 | P1 | 对方有新分享时通知（可关闭） |
| 主题跳过 | P0 | 可以跳过某天不参与 |

### 第一版不做

- ❌ 社区/公开分享
- ❌ 评论/点赞功能
- ❌ 多人/群组
- ❌ 复杂的数据统计
- ❌ 自定义主题（由系统提供）

## 功能详细说明

### 1. 用户注册与配对

#### 1.1 注册流程
```
用户 A 注册 → 生成唯一邀请码 → 分享给用户 B
用户 B 注册 → 输入邀请码 → 配对成功
```

#### 1.2 配对规则
- 一个账号只能与一个账号配对
- 配对后可解绑（需双方确认或等待期）
- 配对关系在数据库中记录

#### 1.3 数据结构
```json
{
  "user_id": "uuid",
  "partner_id": "uuid (nullable)",
  "invite_code": "string",
  "created_at": "timestamp"
}
```

---

### 2. 每日主题系统

#### 2.1 主题展示
- 每天 00:00 刷新主题（用户时区）
- 主题显示在首页顶部
- 可点击查看历史主题

#### 2.2 主题类型示例

**日常类：**
| 主题 | 说明 |
|------|------|
| 此刻你在... | 最开放的主题，任何场景都可以 |
| 今天的午餐 | 食物分享 |
| 窗外的风景 | 环境分享 |
| 此刻的心情 | 选 emoji + 可选文字 |
| 今天的小确幸 | 积极情绪分享 |

**回忆类：**
| 主题 | 说明 |
|------|------|
| 今天想起我们的... | 触发共同回忆 |
| 想和你一起做的... | 未来期待 |

**趣味类：**
| 主题 | 说明 |
|------|------|
| 如果现在在一起，我们会... | 想象场景 |
| 猜猜对方此刻在... | 互动猜测（可选功能） |

#### 2.3 主题库管理
- 第一版：预设 20-30 个主题循环
- 后续：可根据用户反馈增加/调整

---

### 3. 上传分享功能

#### 3.1 上传方式
```
选择今日主题 → 点击上传 → 选择照片（可选）→ 添加文字（可选）→ 发布
```

#### 3.2 内容要求
| 内容类型 | 必填 | 说明 |
|----------|------|------|
| 照片 | 否 | 最多 5 张，支持常见格式 |
| 文字 | 否 | 最多 200 字 |
| 表情 | 否 | 快速选择心情 |

**最小化要求：** 至少上传照片或文字其中之一

#### 3.3 上传时间
- 当天任意时间可上传
- 无截止时间
- 可补传（但标记为补传）

---

### 4. 时间轴展示

#### 4.1 展示逻辑
```
按日期倒序排列
每天显示：
  - 日期 + 主题
  - 双方的分享（并排展示）
  - 如果只有一方上传，也正常显示
```

#### 4.2 界面布局（移动端优先）
```
┌─────────────────────────┐
│ 2026-03-23 周一         │
│ 今日主题：此刻你在...   │
├─────────────────────────┤
│ ┌─────────┬─────────┐   │
│ │  我的   │  TA 的   │   │
│ │  [照片] │  [照片] │   │
│ │  文字   │  文字   │   │
│ │ 12:30   │ 14:15   │   │
│ └─────────┴─────────┘   │
├─────────────────────────┤
│ 2026-03-22 周日         │
│ ...                     │
└─────────────────────────┘
```

#### 4.3 特殊情况处理
| 情况 | 展示方式 |
|------|----------|
| 只有一方上传 | 正常显示，不标记"缺失" |
| 双方都未上传 | 该日期不显示或显示"跳过" |
| 补传 | 标记"补传于 X 月 X 日" |

### 6. 隐私与安全

#### 6.1 数据可见性
- 所有分享默认仅配对双方可见
- 第一版不提供公开/分享功能

#### 6.2 账号安全
- 密码加密存储
- 支持邮箱验证
- 支持找回密码

## 用户流程

### 完整用户旅程

```
1. 访问网站 → 注册账号
2. 获取邀请码 → 分享给伴侣
3. 伴侣注册并输入邀请码 → 配对成功
4. 进入首页 → 查看今日主题
5. 点击上传 → 添加照片/文字 → 发布
6. 查看时间轴 → 看到对方的分享
7. （可选）收到对方新分享的通知
```

### 核心页面

| 页面 | 功能 |
|------|------|
| 首页/时间轴 | 展示历史分享 + 今日主题 |
| 上传页 | 上传照片/文字 |
| 个人中心 | 账号设置、配对管理、通知设置 |
| 登录/注册页 | 基础账号功能 |

## 技术架构建议

### 推荐技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | React + Tailwind CSS | 响应式设计，快速开发 |
| 后端 | Node.js (Express/Nest) | 与前端技术栈统一 |
| 数据库 | PostgreSQL / SQLite | 关系型，支持配对关系 |
| 存储 | 云存储（AWS S3/阿里云 OSS） | 图片存储 |
| 部署 | Vercel (前端) + Railway (后端) | 免费额度足够 MVP |

### 核心数据表设计

```sql
-- 用户表
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  invite_code VARCHAR(50) UNIQUE NOT NULL,
  partner_id UUID REFERENCES users(id),
  created_at TIMESTAMP DEFAULT NOW()
);

-- 分享表
CREATE TABLE posts (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id),
  date DATE NOT NULL,
  theme VARCHAR(255) NOT NULL,
  image_url VARCHAR(500),
  text TEXT,
  is_late_post BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(user_id, date)
);

-- 主题表
CREATE TABLE themes (
  id UUID PRIMARY KEY,
  text VARCHAR(255) NOT NULL,
  category VARCHAR(50),
  is_active BOOLEAN DEFAULT TRUE
);
```

## 成功指标

### MVP 阶段核心指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 配对成功率 | > 60% | 注册后成功邀请伴侣的比例 |
| 7 日留存 | > 40% | 配对后 7 天仍活跃的比例 |
| 周均分享次数 | > 3 次 | 每对情侣每周分享次数 |
| 双方参与率 | > 50% | 双方都有上传的日期占比 |

### 用户反馈关注点

- "感觉有压力吗？"
- "愿意继续使用吗？"
- "会推荐给朋友吗？"

## 风险与应对

| 风险 | 可能性 | 应对措施 |
|------|--------|----------|
| 一方不参与，另一方流失 | 高 | 产品设计上允许单人记录，不强调"缺失" |
| 新鲜感过后使用频率下降 | 中 | 主题多样化，保持趣味性 |
| 隐私担忧 | 中 | 明确隐私政策，技术实现上保障 |
| 被用于非情侣场景 | 低 | 产品定位清晰，但不限制使用场景 |
